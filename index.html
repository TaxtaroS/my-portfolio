Situation · 내가 마주한 문제

분석 결과 전환 오류

LLM 분석 결과가 생성되었지만 중간 grounding 단계에서 걸리면서 화면에는 로컬 분석 결과가 표시되는 문제가 있었습니다. 사용자가 기대한 결과와 실제 표시 결과가 달라지는 지점을 추적했습니다.

PDF 재구성 노이즈

HWP/HWPX를 파싱하고 정규화한 뒤 PDF로 다시 만드는 과정에서 불용어, 이미지, 표 데이터가 함께 노출되었습니다. 비교 화면에 보여줄 텍스트와 보관할 OCR 데이터를 분리할 필요가 있었습니다.

배포와 비교 기능 오류

EC2 배포 후 로그인 해시 문제가 발생했고, 문서 비교에서는 로컬 방식만으로 유사도를 안정적으로 계산하지 못했습니다. 기능이 실패할 때 사용자에게 어떤 상태를 보여줄지도 함께 고민했습니다.

Tech · 왜 이 기술을 사용했는가

React, Vite, TypeScript

문서 업로드, 분석 결과, 비교 화면처럼 상태 변화가 많은 UI를 빠르게 구성하기 위해 사용했습니다. TypeScript로 API 응답과 화면 데이터의 형태를 맞추며 연결 오류를 줄였습니다.

Python, FastAPI

HWP/HWPX 파싱, 텍스트 정규화, LLM 요청처럼 문서 처리와 AI 연동이 많은 작업에 Python을 사용했습니다. FastAPI로 분석, 인증, 문서 비교 API를 나누어 프론트엔드와 연결했습니다.

MongoDB, AWS, Vercel, Docker

사용자 정보와 분석 결과를 저장하기 위해 MongoDB를 사용했고, 프론트엔드는 Vercel, 백엔드는 AWS EC2로 배포했습니다. Docker와 GitHub로 실행 환경과 협업 흐름을 맞췄습니다.

Action · 내가 수행한 개선

01

로컬 분석과 LLM 분석이 전환되는 조건을 확인하고, 과도하게 막히던 grounding 검증을 보류하는 방향으로 파이프라인을 단순화했습니다.

02

OCR로 잡히는 이미지성 데이터는 따로 보관하고, 실제 텍스트만 정규화해 PDF를 생성하도록 문서 변환 흐름을 정리했습니다.

03

EC2 배포 환경에서 HTTP를 HTTPS로 전환하고, Vercel에서 사용한 해싱 방식을 참고해 토큰 재사용 위험을 줄였습니다.

04

문서 유사도 비교는 OpenAI API 기반 청킹 결과를 활용해 비교 가능하도록 유도하고, 비교가 불가능한 경우에는 상태를 명확히 표시했습니다.

Result · 결과와 회고

  • 분석 결과 표시, PDF 변환, EC2 배포 환경 로그인 문제는 정상 동작하도록 해결했습니다.
  • 문서 유사도 비교는 OpenAI API를 활용한 방식으로 일부 개선했지만, 로컬 fallback까지 완전히 해결하지는 못했습니다.
  • 해결하지 못한 경우에도 빈 화면 대신 비교 불가 상태를 보여주도록 예외 처리를 적용했습니다.
  • 생성형 AI는 속도를 높여 주지만, 실제 서비스에 넣기 전에는 직접 검증하고 의심하는 태도가 필요하다는 점을 배웠습니다.

프롤로그 · 오래된 꿈

군대에 가기 전, 저는 동생들과 “우리들만의 리니지”를 꿈꾸던 시절이 있었습니다. 누군가는 기획을 하고, 누군가는 그림을 그리고, 누군가는 이미지를 만들며 게임을 이야기했습니다. 저도 막연히 게임 개발자를 꿈꿨습니다. 하지만 그때의 저는 꿈보다 생계를 먼저 바라봐야 했고, 결국 다른 길로 걸어갔습니다.

1장 · 현장에서 배운 트러블슈팅

그 길이 완전히 돌아가는 길이었다고 생각하지는 않습니다. 건설 현장과 태양광 회사, 의료기기 제조 현장에서 저는 문제를 몸으로 배웠습니다. 설비가 멈추면 어디서부터 봐야 하는지, 문서가 흩어지면 어떻게 정리해야 하는지, 사람이 바뀌어도 일이 이어지려면 어떤 기록이 필요한지 배웠습니다. 특히 스마트공장 구축 경험은 제게 큰 전환점이었습니다. 하드웨어가 좋아지는 것만으로는 충분하지 않았고, 현장을 이해하지 못한 소프트웨어는 결국 사람에게 쓰이지 못한다는 사실을 보았습니다.

개발을 배우며 저는 다시 같은 즐거움을 느끼고 있습니다. 애플리케이션을 만들다 보면 수많은 에러를 만납니다. 처음에는 막막하지만, 에러 메시지를 읽고 원인을 좁히고 조치를 취해 해결하는 과정이 점점 재미있어졌습니다. 수업 중 모델 서빙을 특히 재미있게 느꼈고, API와 모니터링에도 자연스럽게 관심이 생겼습니다. 사용자의 요청에 빠르게 반응하고, 문제가 생기면 원인을 찾아 다시 살리는 일. 어쩌면 저는 오래전부터 트러블슈팅을 좋아하는 사람이었는지도 모릅니다.

2장 · 기록하는 사람
